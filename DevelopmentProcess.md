# My Development Process

This document describes how I work across all my open-source projects (the `phax` ecosystem).
It is the operational counterpart to [CodingStyleguide.md](CodingStyleguide.md) -- that file says *how the code should look*;
this file says *how the work gets done* from a fresh machine to a released artifact.

The whole process is built around a single assumption:
**every project is checked out as a sibling directory under one common parent folder** (e.g. `~/dev/git/`).
All cross-cutting scripts live in that parent folder and operate on all sibling repos.
This `meta` project is the single source of truth for the active project list (see `EProject`) and generates the scripts and aggregate documentation.

## Workstation Setup

* JDK 17 (active development baseline; CI also builds against 21 and 25)
* Apache Maven 3.x (see [MavenCheatSheet.md](MavenCheatSheet.md) for `settings.xml`)
* Git (see [GitCheatSheet.md](GitCheatSheet.md) for global config)
* Eclipse IDE -- use the formatter, preferences and dictionary checked into this repo:
  * `eclipse-formatter-ph.xml` -- my personal formatter
  * `eclipse-formatter-google.xml` -- alternative
  * `eclipse-preferences-2026-03.epf` -- exported workspace preferences
  * `eclipse-dictionary.dict` -- spell-check dictionary
* A working PGP key (see [MavenPublication.md](MavenPublication.md)) for signing releases to Maven Central

## Initial Bootstrap

From an empty parent folder:

1. Clone `meta` first:

   ```shell
   git clone https://github.com/phax/meta
   ```

2. Run the generated clone script that ships with the parent folder (or grab it from this repo's `generated/` tree):

   ```shell
   ./git_clone.sh
   ```

   This skips any repo whose directory already exists, so it is safe to re-run.

3. Build everything once, in order, to populate the local Maven repository:

   ```shell
   ./mvn_clean_install.sh
   ```

   Pass extra Maven arguments through `$@`, e.g. `./mvn_clean_install.sh -DskipTests` (or use the `_skip_tests` variant).

## The `EProject` Registry

`com.helger.meta.project.EProject` is the central enum of all active projects. Each entry carries:

* repo name, optional sub-module, Maven coordinates
* owner (e.g. `phax`, `austriapro`)
* project type (library, Maven plugin, parent POM, ...)
* current version
* required JDK level
* whether GitHub Pages / Wiki are enabled

Deprecated projects move to `EProjectDeprecated`. Externally-defined custom projects come in via XML and are merged through `ProjectList`.

**Order matters.** The enum is declared in build order (parent POMs first, then leaves). The generated `git_clone.sh` and `mvn_clean_install.sh` walk projects in that order so a clean local build resolves bottom-up.

## Adding a New Module

1. Add the entry to `EProject` (or `EProjectDeprecated` when retiring something), placed at the correct position in the dependency order.
2. If it introduces a new external dependency, register it in `EExternalDependency`.
3. Re-run the relevant generators (see *Meta Tools* below) to refresh:
   * The shell scripts in the parent folder
   * `README.md` (the project listing on this page)
   * `LinesOfCode.md` / `stats.md`
4. Commit the regenerated artifacts together with the `EProject` change.

## Per-Project Required Files

`MainCheckProjectRequiredFiles` enforces that each sibling repo carries the conventional set of files (license header template, `findbugs-exclude.xml`, GitHub Actions workflow, README badges, etc.). Run it after onboarding a new project, and again whenever a convention changes across the ecosystem so existing repos can be brought into line.

Other checkers in `com.helger.meta.tools.buildsystem`:

| Tool | Purpose |
| --- | --- |
| `MainCheckPOMArtifactVersions` | POM `<version>` values match `EProject` |
| `MainCheckGitHubActionVersions` | `actions/*@vN` versions are consistent across all `maven.yml` |
| `MainCheckFindBugsExcludes` | `findbugs-exclude.xml` is present and conventional |
| `MainCheckProjectRequiredFiles` | License headers, `.github/workflows/maven.yml`, etc. |
| `MainCreateMetaREADME` | Regenerate `README.md` from `EProject` |
| `MainUpdateREADMEBadges` / `MainUpdateREADMEFooter` | Refresh badge and footer blocks in each repo's `README.md` |
| `MainCreateMetaLinesOfCode` | Regenerate `LinesOfCode.md` / `stats.md` |
| `MainUpdateLicenseTemplateYear` | Bump the copyright year in every repo's license template |
| `MainCreateKnownDependencyPOM` | Emit a synthetic POM listing all known external deps |
| `MainCreateRepoList` | Emit the textual repo list |
| `MainExtractParentPOMDeps` | Pull dependency declarations out of parent POMs |
| `MainUpdateJavadocCSS` | Refresh the bundled Javadoc CSS |
| `MainEclipseCompilerErrorsSetDefault` / `MainEclipseCompilerErrorsRestoreBackup` | Toggle Eclipse compiler-warning settings across repos |

Other tool groups:

* `com.helger.meta.tools.cmdline` -- `MainCreateShellScripts` regenerates the parent-folder scripts; `MainCreateBatchFiles` and `MainCreateBuildAllPOM` emit aggregate driver artifacts.
* `com.helger.meta.tools.codingstyleguide` -- ASM-based bytecode/source checks that validate naming, formatting and structural conventions across compiled classes.
* `com.helger.meta.tools.wsdlgen` -- consumes `.interface` files and emits WSDL/XSD into `generated/`.
* `com.helger.meta.tools.translation` -- extracts translatable strings.
* `com.helger.meta.tools.sysinfo` -- environment introspection.

All tools are `public static void main` entry points; none of them are JUnit tests. Run from the IDE or with `mvn exec:java -Dexec.mainClass=...`.

## Continuous Integration

Each repo carries `.github/workflows/maven.yml`:

* Matrix builds against Java 17, 21 and 25 on `ubuntu-latest`
* Maven local repo is cached by `pom.xml` hash
* On Java 17 only: `mvn -P release-snapshot deploy` publishes a snapshot to Maven Central, using `MAVEN_USERNAME` / `MAVEN_PASSWORD` from repository secrets
* On Java 21 and 25: `mvn install` for verification only

`MainCheckGitHubActionVersions` keeps the `actions/*@vN` pins aligned across every repo's workflow file.

## Releasing to Maven Central

Per-project release uses `maven-release-plugin`:

```shell
mvn release:prepare
mvn release:perform
```

A wrapper batch is in [MavenCheatSheet.md](MavenCheatSheet.md). Artifacts are signed with the PGP key documented in [MavenPublication.md](MavenPublication.md) (fingerprint `670B E7D0 76A0 C89F 5F07 625A E574 28DA 9E87 9E7D`).

After releasing, bump the version in `EProject` to the next SNAPSHOT and regenerate `README.md` so the public version listing reflects reality.

## Housekeeping (Periodic)

* `MainUpdateLicenseTemplateYear` once per year, then `./mvn_license_format.sh` to roll the header into every source file.
* `./git_gc.sh` / `./git_gc_auto.sh` to compact repositories.
* `./git_normalize_crlf.sh` after CRLF-related changes on Windows.
* Re-run the `MainCheck*` tools whenever a convention changes, so all sibling repos catch up uniformly.
